# 体育博彩平台盘口交互设计方案

## 1. 核心设计原则

本方案基于对不同用户群体行为的深入洞察，旨在提供一个既对新手友好，又为资深玩家提供足够策略深度的交互模型。其核心原则如下：

### 1.1. 允许在选项 > 2 的盘口中进行多选

对于拥有三个或更多选项的盘口，允许多选是满足用户正常投注行为的必要功能。

- **场景示例：**
  - **1x2 (独赢盘):** 用户选择主胜（1）和客胜（2）两个选项，实质上是构建了一个“分头投注”（Dutching）策略的雏形，或是为后续构建“双重机会”（Double Chance）盘口做准备。
  - **冠军/优胜者 (Outright):** 在一个有20个选项的冠军盘口中，用户可能希望选择3-4个最有可能的获胜者来分散风险。多选是实现此策略的基础。

- **设计理由：** 这种多选行为是用户降低风险、构建组合策略的自然延伸，平台应予以支持，并将复杂性收敛在投注单（Bet Slip）中进行处理，而非在主盘口列表上造成困扰。

### 1.2. 不建议在选项 = 2 的盘口中进行多选

对于只有两个对立选项的盘口（例如大小球、亚洲盘、让分盘），允许多选不仅没有实际价值，还会带来负面影响。

- **场景示例：**
  - **大小球 (Over/Under 2.5):** 用户同时选择“大2.5”和“小2.5”。

- **设计理由：**
  - **操作冗余对新手不友好：** 新手可能会误触并选择两个选项，造成困惑。由于平台赔率设计决定了同时投注两个对立选项必然亏损（抽水/juice），这种操作没有任何收益。
  - **对专家无意义：** 资深玩家或套利者（Arber）非常清楚平台内的赔率无法构成套利空间。他们的套利行为必然是跨平台（cross-platform）进行的。因此，在平台内部提供这种多选功能，对于吸引或服务这部分用户群体并无帮助。

## 2. 高级用户需求分析

在核心原则之上，我们识别出两种关键的高级用户需求，它们是驱动多选交互设计的核心动力。

### 2.1. 滚球中的赔率监控与机会捕捉

资深玩家在进行滚球（Live Betting）投注时，其行为模式并非简单的“选择-下注”。他们经常需要同时监控同一个盘口中多个选项的赔率动态变化。例如，在一场足球比赛的最后10分钟，他们可能同时关注“小2.5球”和“平局”两个选项的赔率，等待其中一个达到他们认为有价值的“价位”时迅速出手。当前“点击即替换”的单选模式无法满足这种并列观察、快速切换的需求。

### 2.2. 过关投注中的复杂选项组合 (Ways)

在构建过关或系统投注（Parlay/System Bet）时，用户有时希望将一个“投注项”（leg）定义为“在某个盘口中，A或B发生即可”。例如，一个3/2的过关投注（AB[C/D]），其中前2关是2场不同比赛的胜负盘，而第3关用户希望是“比赛D的主队胜或客队胜”(最终生成AB,AC,AD,BC,BD5串)。这种将同一盘口的多个选项打包成一个“路”（Way）的需求，在标准交互中难以实现。

## 3. 交互设计方案：以投注单为核心的解决方案

为了同时满足界面简洁性与高级用户的功能需求，我们不改变盘口列表的默认单选交互，而是将所有复杂性管理和高级功能全部收敛到**投注单 (Bet Slip)** 中。投注单将根据用户添加的选项，动态切换其模式和功能。

### 3.1. 盘口列表交互 (Market List Interaction)

- **默认行为：** 保持单选。用户点击任何一个盘口的选项，该选项被添加至投注单。
- **触发多选：** 当用户点击**同一个盘口**的**第二个或更多选项**时，系统不替换投注单中的已有选项，而是将新选项**追加**到投注单中。此时，投注单将自动识别来自同一盘口的多个选项，并进入特殊处理模式。

### 3.2. 投注单交互 (Bet Slip Interaction)

投注单是本方案的核心，它将根据自身内容和用户选择的模式（单注/过关）呈现不同的功能。

#### 3.2.1. 单注模式 (Single Bet Mode) - 赔率监控工作台

此模式主要服务于滚球场景下的高级用户，将投注单变为一个赔率监控和快速交易的工具。

- **触发条件：** 用户处于“单注”模式，且向投注单中添加了来自**同一个滚球盘口**的多个选项。
- **界面表现：**
  - 投注单中对应的投注卡片（Slip Card）会扩展，**并列显示所有被选中选项的名称和实时变化的赔率**。
  - 用户可以清晰地看到 “小2.5” @ 1.90, “平局” @ 3.50, “主胜” @ 2.10 在同一个卡片内同时跳动。
  - 卡片上会有一个明确的切换控件（例如 Radio Button），允许用户在这些并列的选项中**指定当前要投注的目标**。
  - 金额输入框和“投注”按钮将只对当前指定的目标生效。

- **用户价值：** 用户无需离开投注单，即可实现对多个潜在投注目标的并行观察，并在赔率达到心理价位时，通过“切换-输入金额-投注”的简单操作，在2-3秒内完成投注，完美满足了滚球场景下的专业需求。

#### 3.2.2. 过关模式 (Parlay/System Mode) - “Ways”组合构建器

此模式用于满足用户在构建复杂过关投注时，希望将同一盘口的多个结果打包为一个投注项的需求。

- **触发条件：** 用户处于“过关/系统”模式，且向投注单中添加了来自**同一个盘口**的多个选项。
- **界面表现：**
  - 默认情况下，这多个选项被视为独立的投注项，如果平台规则不允许（例如，同一比赛的胜与负不能串关），则会提示冲突。
  - **关键设计：** 在这张投注卡片的右上角或工具栏中，出现一个**“转化为Ways” (Combine as a 'Way')** 的按钮或开关。
  - 当用户点击此按钮后，这张卡片内的多个选项会 并列展示，选中的选项才会高亮

- **用户价值：** 极大地增强了过关投注的策略灵活性。用户可以精确地构建“某队不败”、“比赛分出胜负”等自定义的投注条件，并将其无缝融入到更复杂的系统投注中，满足了最高阶玩家的策略构建需求。

---

## 4. 方案总结

该设计方案通过保持盘口列表的简洁性，并将所有高级功能和复杂性移至投注单内部进行智能处理，成功地实现了对新手用户和专业玩家的双重友好。

| 需求 | 解决方案 |
| :--- | :--- |
| **基础投注** | 盘口列表保持单选，点击即添加，简单直观。 |
| **风险对冲 (双重机会)** | 在盘口列表点击同一盘口的多个选项，投注单自动准备好“Ways”合并的基础。 |
| **滚球赔率监控** | 投注单的“单注模式”提供多选项赔率并列展示和快速切换功能。 |
| **复杂过关 (Ways)** | 投注单的“过关模式”提供“合并为Ways”功能，允许用户动态创建自定义投注项。 |

这种分层式的交互设计，确保了功能强大的同时，不会牺牲核心流程的简洁和易用性，是平衡不同用户群体需求的最佳实践。
